# cc-bash-proxy

Policy-only proxy for CLI invocations used by AI agents and shell hooks.

This README documents current user-facing behavior. Planned or proposed
behavior belongs in [`docs/dev/spec`](docs/dev/spec/README.md) and must not be
treated as shipped unless the spec status is `implemented` and the behavior is
covered by tests.

`cc-bash-proxy` does not rewrite commands. It parses the input command string,
evaluates permission policy, and returns `allow`, `ask`, or `deny`. The command
string passed through the hook remains the original command.

Parser-backed normalization is evaluation-only:

- `bash -c 'git status'` and supported shell-wrapper forms are evaluated by
  parsing the inner command.
- `/usr/bin/git status` matches `command.name: git`, while the original program
  token remains available in trace/debug output.
- `aws --profile myprof sts get-caller-identity`,
  `aws --profile=myprof sts get-caller-identity`, and
  `AWS_PROFILE=myprof aws sts get-caller-identity` expose the same AWS semantic
  `profile`, `service`, and `operation` fields for permission matching.

## Current Status

Stable user-facing features:

- Claude Code hook integration via `cc-bash-proxy hook`
- permission buckets `deny`, `ask`, and `allow`
- permission rules using `command`, `env`, and `patterns`
- command-specific semantic parsers for higher-risk tools such as `git`, `gh`,
  `aws`, `kubectl`, and `helmfile`
- verified hook artifacts generated by `cc-bash-proxy verify`
- `doctor`, `verify`, `init`, `version`, `hook`, and `semantic-schema`

Top-level `rewrite`, rewrite primitives, rewrite-local tests, permission
`match`, and singular `pattern` are not supported. If a config contains
top-level `rewrite`, `verify` fails with migration guidance.

## Typical Workflow

```sh
cc-bash-proxy init
$EDITOR ~/.config/cc-bash-proxy/cc-bash-proxy.yml
cc-bash-proxy verify
```

Register Claude Code with a `PreToolUse` Bash hook:

```json
{
  "matcher": "Bash",
  "hooks": [
    { "type": "command", "command": "cc-bash-proxy hook" }
  ]
}
```

By default, `cc-bash-proxy hook` fails closed when the verified artifact is
missing or stale. Run `cc-bash-proxy verify` after editing policy. The
`--auto-verify` flag regenerates artifacts during hook execution and should be
used only when that review tradeoff is acceptable.

## Permission Model

Permission evaluation is deterministic:

1. parse the original command string into a `CommandPlan`
2. evaluate `deny`
3. evaluate `ask`
4. evaluate `allow`
5. return `ask` for unsafe or ambiguous command shapes that are not explicitly
   denied

Each rule uses `command`, `env`, or `patterns`. `command` and `patterns` cannot
be combined in one rule. `env` can be combined with either.

```yaml
permission:
  deny:
    - name: git force push
      command:
        name: git
        semantic:
          verb: push
          force: true
      message: "force push is blocked"

  allow:
    - name: aws identity
      command:
        name: aws
        semantic:
          service: sts
          operation: get-caller-identity
      env:
        requires:
          - AWS_PROFILE
      message: "allow AWS identity check"

  ask:
    - name: helm upgrade fallback
      patterns:
        - "^helm\\s+upgrade\\b"
      env:
        requires:
          - KUBECONFIG
      message: "helm upgrade requires confirmation"
```

Use `patterns` for commands without a semantic parser, including read-only
basics such as `ls`, `cat`, `grep`, `head`, `tail`, and `pwd`:

```yaml
permission:
  allow:
    - name: read-only shell basics
      patterns:
        - "^ls(\\s|$)"
        - "^pwd$"
        - "^cat\\s+[^;&|`$()]+$"
        - "^head\\s+"
        - "^tail\\s+"
```

## AWS Style

Preferred AWS command style in user and project guidance:

```sh
AWS_PROFILE=myprof aws eks list-clusters
```

Non-preferred style:

```sh
aws --profile myprof eks list-clusters
```

`cc-bash-proxy` does not convert between these forms. It parses AWS
`profile`, `service`, and `operation` semantically for policy evaluation. Use
policy `ask` or `deny` rules for ambiguous or dangerous command styles.

`rtk` integration is outside this policy model. When `--rtk` is used, permission
evaluation still happens before `rtk` is invoked.

## Shell Evaluation

Supported shell `-c` wrappers are evaluated by parsing the inner command for
permission decisions:

- `bash -c 'git status'`
- `sh -c 'git status'`
- `/bin/bash -c 'git status'`
- `env bash -c 'git status'`
- `command bash -c 'git status'`
- `sudo -u root bash -c 'git status'`
- `timeout 10 bash -c 'git status'`
- `busybox sh -c 'git status'`

This is not command rewriting. The command returned to the hook remains the
original command. Non-`-c` shell invocations such as `bash script.sh` are not
expanded. Unsafe or unparsable inner shell commands fail closed.
