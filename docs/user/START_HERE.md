# Start Here

`cc-bash-proxy` is a local CLI that sits in front of command execution and
evaluates permission policy. It parses the original command string, decides
`allow`, `ask`, or `deny`, and does not rewrite the command that will run.

## Quick Start

```sh
cc-bash-proxy init
$EDITOR ~/.config/cc-bash-proxy/cc-bash-proxy.yml
cc-bash-proxy verify
cc-bash-proxy doctor --format json
```

Then register `cc-bash-proxy hook` in Claude Code.

## Claude Code

Add `cc-bash-proxy hook` as a `PreToolUse` Bash hook.

```json
{
  "matcher": "Bash",
  "hooks": [
    { "type": "command", "command": "cc-bash-proxy hook" }
  ]
}
```

The hook returns:

- `allow`: auto-allow immediately
- `ask`: let Claude prompt
- `deny`: block immediately

The hook trusts verified artifacts generated by `cc-bash-proxy verify`. By
default it denies when that artifact is missing or stale. `--auto-verify`
regenerates the artifact at hook time and should be used only when that review
tradeoff is acceptable.

If you also use `rtk`, prefer `cc-bash-proxy hook --rtk` instead of registering
multiple Bash hooks. Permission evaluation happens before `rtk` is invoked.
`rtk` integration is outside the permission-only policy model.

## Current Config Model

- top-level keys are `permission`, `test`, and optional
  `claude_permission_merge_mode`
- top-level `rewrite` is not supported
- permission rules use only `command`, `env`, and `patterns`
- singular `pattern` and permission `match` are not supported
- permission buckets are evaluated in the order `deny -> ask -> allow`
- top-level `test` is for end-to-end behavior

Parser-backed normalization is evaluation-only:

- shell `-c` wrappers are evaluated by parsing the inner command
- absolute command paths match by basename
- AWS `--profile` and `AWS_PROFILE` are parsed as semantic `profile`

## AWS Style

Preferred:

```sh
AWS_PROFILE=myprof aws eks list-clusters
```

Non-preferred:

```sh
aws --profile myprof eks list-clusters
```

`cc-bash-proxy` does not convert between these forms. It evaluates AWS
`profile`, `service`, and `operation` through the semantic parser. Use policy
`ask` or `deny` for ambiguous or dangerous command styles.
